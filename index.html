Unit Testing in Java

Master JUnit, TestNG, and Mockito — explained simply with real code examples, interview tips, and comparisons.

📖
What is Unit Testing?
The foundation of reliable Java code

🔍 Simple Definition

Unit Testing means testing the smallest piece of code (a single method or class) in isolation to make sure it works correctly. You write code that tests your code.

💡 Real-world analogy: Think of unit testing like testing each ingredient before cooking a dish. You taste the salt, the sugar, the flour — individually. That's unit testing. Tasting the final dish is integration testing.

✅ Why Write Tests?

• Catch bugs early (cheaper to fix)
• Confidence when refactoring code
• Acts as live documentation
• Required in most companies

🧱 What is a "Unit"?

A unit is typically:
• A single method
• A single class
It should NOT depend on a database, API, or file system.

🏗️ The AAA Pattern — Every Test Follows This

CalculatorTest.java 
@Test
void shouldAddTwoNumbers() {

    // ARRANGE — set up your data
    Calculator calc = new Calculator();

    // ACT — call the method you're testing
    int result = calc.add(3, 4);

    // ASSERT — verify the result
    assertEquals(7, result);
}

🛠️ The 3 Main Tools in Java

JUnit 5 Most Popular

The standard testing framework for Java. Used by almost every Java project. Provides @Test, assertions, lifecycle hooks.

TestNG

A powerful alternative to JUnit with extra features like test groups, parallel execution, and data providers (great for enterprise projects).

Mockito Used WITH JUnit/TestNG

A mocking library. When your class depends on a database or an API, Mockito creates a fake version of that dependency so you can test your class in isolation.

🔴
JUnit 5
The most widely-used Java testing framework

What is JUnit?

JUnit is a testing framework for Java. It lets you write test methods using annotations like @Test, provides assertion methods like assertEquals, and runs your tests automatically (e.g., in Maven/Gradle or your IDE).

📦 Maven Dependency

pom.xml 
<dependency>
    <groupId>org.junit.jupiter</groupId>
    <artifactId>junit-jupiter</artifactId>
    <version>5.10.0</version>
    <scope>test</scope>
</dependency>

🏷️ Key JUnit 5 Annotations

@Test

Marks a method as a test case. JUnit will run it automatically.

@BeforeEach

Runs before EACH test method. Good for setup (creating objects).

@AfterEach

Runs after EACH test method. Good for cleanup.

@BeforeAll

Runs ONCE before all tests in the class. Must be static.

@AfterAll

Runs ONCE after all tests in the class. Must be static.

@Disabled

Skips the test. Like commenting it out but keeps it visible.

@DisplayName

Gives a human-readable name to the test for reports.

@ParameterizedTest

Runs the same test with multiple inputs.

✅ Assertions (How to check results)

Assertions cheatsheet 
// Check equality
assertEquals(10, result);

// Check boolean
assertTrue(result > 0);
assertFalse(list.isEmpty());

// Check null
assertNotNull(user);
assertNull(deletedUser);

// Check exception is thrown
assertThrows(IllegalArgumentException.class, () -> {
    calc.divide(10, 0);
});

// Check multiple assertions together
assertAll(
    () -> assertEquals("Ankush", user.getName()),
    () -> assertEquals(25, user.getAge())
);

🔢 Parameterized Test Example

Run same test with multiple inputs 
@ParameterizedTest
@ValueSource(ints = {2, 4, 6, 8})
void shouldBeEvenNumber(int number) {
    assertTrue(number % 2 == 0);
}
// Runs 4 times: once for 2, 4, 6, 8

📋 Complete Example — Calculator Test

CalculatorTest.java 
import org.junit.jupiter.api.*;
import static org.junit.jupiter.api.Assertions.*;

class CalculatorTest {

    private Calculator calc;

    @BeforeEach
    void setUp() {
        calc = new Calculator(); // runs before each test
    }

    @Test
    @DisplayName("3 + 4 should equal 7")
    void testAdd() {
        int result = calc.add(3, 4);
        assertEquals(7, result);
    }

    @Test
    void testDivideByZeroThrowsException() {
        assertThrows(ArithmeticException.class,
            () -> calc.divide(10, 0));
    }

    @Test
    @Disabled("Feature not implemented yet")
    void testSqrt() {
        // This test will be skipped
    }
}
🔵
TestNG
Enterprise-grade testing — parallel, grouped, data-driven

What is TestNG?

TestNG (Test Next Generation) is an alternative to JUnit with more powerful features like test grouping, parallel execution, and data providers. It's very popular in enterprise Java and Selenium test automation projects.

💡 Key difference: TestNG uses testng.xml to configure which tests run, their order, and grouping — JUnit doesn't have this central config file.

🏷️ Key TestNG Annotations (vs JUnit)

Purpose JUnit 5 TestNG
Mark a test@Test@Test
Before each test@BeforeEach@BeforeMethod
After each test@AfterEach@AfterMethod
Before all tests@BeforeAll@BeforeClass
After all tests@AfterAll@AfterClass
Skip a test@Disabled@Test(enabled=false)
Multiple inputs@ParameterizedTest@DataProvider
Group tests❌ Not built-in@Test(groups=...) ✅
Parallel testsLimited✅ Built-in via XML

🔑 Unique TestNG Features with Examples

1️⃣ Test Groups

Group tests by category and run only specific groups. E.g., run only "smoke" tests before deployment.

GroupsExample.java 
@Test(groups = "smoke")
void loginTest() { ... }

@Test(groups = "regression")
void paymentTest() { ... }

// In testng.xml — run only smoke tests
<groups>
  <run>
    <include name="smoke"/>
  </run>
</groups>

2️⃣ @DataProvider — Run test with multiple data sets

Like JUnit's @ParameterizedTest but more flexible — can return Objects, not just primitives.

DataProviderExample.java 
@DataProvider(name = "loginData")
public Object[][] getData() {
    return new Object[][] {
        {"admin", "admin123", true},
        {"user1", "wrong",    false},
        {"ankush", "pass",   true}
    };
}

@Test(dataProvider = "loginData")
void testLogin(String user, String pass, boolean expected) {
    boolean actual = authService.login(user, pass);
    assertEquals(expected, actual);
}
// Runs 3 times — once for each row of data

3️⃣ Test Priority & Dependencies

Control the ORDER tests run and make one test depend on another.

PriorityExample.java 
@Test(priority = 1)
void registerUser() { ... }

@Test(priority = 2, dependsOnMethods = "registerUser")
void loginUser() { ... }

@Test(priority = 3, dependsOnMethods = "loginUser")
void placeOrder() { ... }

Mockito
Replace real dependencies with fake ones for isolated testing

The Problem Mockito Solves

Your UserService calls a UserRepository (database). In a unit test, you don't want to actually hit the DB. Mockito creates a fake (mock) UserRepository that returns whatever you tell it to.

💡 Analogy: You're rehearsing a play. The actor playing the villain is sick. You put a cardboard cutout in their place and PRETEND it's them. That's a mock — a fake stand-in that "acts" the way you tell it to.

🔑 3 Core Concepts

1. Mock

A completely fake object. All its methods do nothing / return null by default. You control its behavior.

2. Stub

Telling the mock: "When THIS method is called, return THAT value." Uses when(...).thenReturn(...)

3. Verify

Checking the mock was actually called. "Was save() called exactly once?" Uses verify(...)

4. Spy

A partial mock — wraps a REAL object. Real methods run unless you override them.

📋 Typical Project Structure

UserService.java (the class we're testing) 
public class UserService {

    private final UserRepository userRepo; // dependency

    public UserService(UserRepository userRepo) {
        this.userRepo = userRepo;
    }

    public User getUserById(long id) {
        return userRepo.findById(id)
            .orElseThrow(() -> new RuntimeException("User not found"));
    }

    public void saveUser(User user) {
        userRepo.save(user);
    }
}

✅ Full Mockito Test Example

UserServiceTest.java 
import org.junit.jupiter.api.*;
import org.mockito.*;
import static org.mockito.Mockito.*;
import static org.junit.jupiter.api.Assertions.*;

@ExtendWith(MockitoExtension.class)
class UserServiceTest {

    @Mock
    private UserRepository userRepo; // Mockito creates a fake

    @InjectMocks
    private UserService userService; // injects the mock into service

    @Test
    void shouldReturnUserWhenFound() {
        // ARRANGE — stub the mock
        User fakeUser = new User(1L, "Ankush");
        when(userRepo.findById(1L))
            .thenReturn(Optional.of(fakeUser));

        // ACT
        User result = userService.getUserById(1L);

        // ASSERT
        assertEquals("Ankush", result.getName());
    }

    @Test
    void shouldVerifySaveWasCalled() {
        User user = new User(2L, "Rahul");

        userService.saveUser(user);

        // VERIFY — was save() called exactly once?
        verify(userRepo, times(1)).save(user);
    }

    @Test
    void shouldThrowWhenUserNotFound() {
        when(userRepo.findById(99L))
            .thenReturn(Optional.empty());

        assertThrows(RuntimeException.class,
            () -> userService.getUserById(99L));
    }
}

🔧 Common Mockito Methods

MethodWhat it does
mock(Class.class)Creates a mock object manually
@MockCreates mock via annotation (cleaner)
@InjectMocksInjects mocks into the class under test
when(x).thenReturn(y)Stubbing — when x is called, return y
when(x).thenThrow(e)When x is called, throw exception e
verify(mock).method()Check that method was called
verify(mock, times(2))Check method was called exactly 2 times
verify(mock, never())Check method was NEVER called
any(), anyString()Argument matchers — match any value

✅ Pro Tip: Use @ExtendWith(MockitoExtension.class) on your test class — this automatically initializes all @Mock and @InjectMocks fields without needing MockitoAnnotations.openMocks(this).

🏆 When to Use What?

Use JUnit 5 when...

• You're in a Spring Boot project
• You want the industry standard
• You need simple, clean tests
• You're starting a new project

Use TestNG when...

• You need test grouping (smoke/regression)
• You need parallel test execution
• You're doing Selenium automation
• You're in an enterprise legacy project

Use Mockito when...

• Your class has dependencies (DB, API, services)
• You want to test in isolation without hitting real resources
• Always use this alongside JUnit 5 in Spring Boot projects

Interview Q&A
Most asked questions — crisp, confident answers
  • 1
    What is unit testing?
    Testing individual methods/classes in isolation to verify they work correctly, without depending on real databases or APIs.
  • 2
    What is the difference between JUnit and TestNG?
    Both are testing frameworks. JUnit 5 is the Spring Boot default and simpler. TestNG has extra features like test groups, parallel execution, and @DataProvider. TestNG is popular for Selenium automation.
  • 3
    What is Mockito? What is a mock?
    Mockito is a mocking library. A mock is a fake object that simulates a real dependency (like a DB or API). It lets you test your class in isolation by controlling what the dependency returns.
  • 4
    Difference between @Mock and @InjectMocks?
    @Mock creates the fake dependency. @InjectMocks creates the class you're testing and automatically injects all @Mock fields into it.
  • 5
    What is the AAA pattern?
    Arrange (set up data), Act (call the method), Assert (verify the result). Every test follows this pattern.
  • 6
    What is the difference between a Mock and a Spy?
    A Mock is completely fake — all methods return null/default. A Spy wraps a REAL object — real methods run unless you specifically override them.
  • 7
    What is when().thenReturn()?
    Stubbing. It tells Mockito: "When THIS method is called on the mock, return THIS value." e.g., when(repo.findById(1L)).thenReturn(Optional.of(user))
  • 8
    What annotations does JUnit 5 use for lifecycle?
    @BeforeEach, @AfterEach, @BeforeAll, @AfterAll. BeforeAll/AfterAll must be static since they run once for the entire class.
  • 9
    How do you test that an exception is thrown?
    Use assertThrows(ExceptionClass.class, () -> { methodCall(); }) in JUnit 5. TestNG uses @Test(expectedExceptions = Exception.class).
  • 10
    What is @DataProvider in TestNG?
    A TestNG feature that supplies multiple sets of input data to a test method. The test runs once for each data row. More flexible than JUnit's @ParameterizedTest since it can pass Objects.
